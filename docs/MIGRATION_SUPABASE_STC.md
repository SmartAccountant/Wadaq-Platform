# ملاحظات نقل Supabase → PostgreSQL على STC

## ما يفعله هذا المستودع حالياً

- **الواجهة (Vite):** كانت تخزّن بيانات التطبيق في **المتصفح** (`sessionStorage`) عبر `WadaqCore.js`، وليست مربوطة مباشرة بقاعدة Supabase في الكود.
- **بعد التفعيل:** عند ضبط `VITE_WADAQ_API_URL` تُخزَّن **حسابات المستخدمين** (بريد/كلمة مرور bcrypt، وجوجل) في **PostgreSQL** على خادمكم، وتُدار الجلسة بـ **JWT** من مجلد `server/`.
- **الفواتير والكيانات الأخرى** ما زالت في التخزين المحلي للمتصفح حتى تُبنى واجهات API كاملة للكيانات (مرحلة لاحقة).

## كلمات مرور المستخدمين بعد الاستيراد

- المستخدمون المستوردون من dump قد يكون لديهم **تجزئة كلمات مرور بصيغة Supabase/Auth** وليست `bcrypt` كما يستخدمها `server/`.
- الحلول العملية: إعادة تعيين كلمات المرور، أو سكربت تحويل لاحق يطابق الصيغة.

## الترتيب الموصى به

1. تشغيل Docker + PostgreSQL على STC (`infra/stc-cloud/`).
2. `pg_dump` من Supabase ثم `pg_restore` على السيرفر (انظر `infra/stc-cloud/README.md`).
3. تطبيق `server/migrations/001_init.sql` إن لم تكن الجداول موجودة، أو دمج المخطط يدوياً مع dump.
4. تشغيل `server/` مع `DATABASE_URL` و`JWT_SECRET` و`GOOGLE_CLIENT_ID`.
5. Nginx أمام API مع HTTPS؛ فتح المنفذ `8787` داخلياً فقط.
6. في Vercel: `VITE_WADAQ_API_URL=https://api.النطاق` (بدون شرطة مائلة أخيرة).
